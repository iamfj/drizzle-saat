## Project overview

`saat` is a TypeScript-native, type-safe database seeding tool for **Drizzle
ORM** ŌĆö fixtures as easy as YAML, deterministic, multi-dialect. It ships as an
ESM **library** (`defineFixture`, `ref`) and a **CLI** (`saat`). It is a
dev/test tool that does wipe-and-reseed; it is **not** for production.

- **Runtime/PM:** Bun (package manager + test runner). Node `>=20` is the
  published target.
- **Language:** TypeScript, ESM, strict mode. Build via tsup. Lint/format via
  Biome.
- **Dialects:** PostgreSQL, MySQL, SQLite ŌĆö behavior must hold for all three.

## Setup & commands

```bash
bun install            # install deps
bun run build          # build library + CLI (tsup)
bun run typecheck      # tsc --noEmit
bun run check          # Biome lint + format check (CI gate)
bun run format         # auto-fix formatting
bun test               # unit + in-process DB tests (SQLite, Postgres)
bun run test:integration   # also runs MySQL integration tests (needs MYSQL_URL)
```

Run a single test file: `bun test test/unit/plan.test.ts`.

Before opening a PR, `bun run check`, `bun run typecheck`, and `bun test` must
all pass.

## Code style

- ESM only. Use **`.js` specifiers** in relative imports (e.g.
  `import { plan } from "./plan.js"`) ŌĆö required by `verbatimModuleSyntax`.
- Use **`import type`** for type-only imports.
- Formatting/linting is enforced by **Biome**: 2-space indent, 100-col width,
  double quotes, semicolons, trailing commas. Don't hand-format ŌĆö run
  `bun run format`.
- All user-facing errors must be `SaatError` subclasses (`src/util/errors.ts`)
  with a clear, actionable message.
- `noExplicitAny` and `noNonNullAssertion` are intentionally off; prefer real
  types but they're allowed where the type system can't follow.

## Testing

- SQLite and Postgres tests run **fully in-process** (`better-sqlite3`,
  `@electric-sql/pglite`) ŌĆö no Docker. They run on every `bun test`.
- MySQL integration tests are gated behind `INTEGRATION=1` and need a reachable
  MySQL via `MYSQL_URL`.
- When adding behavior that branches per dialect, add coverage for **all three**.
- Type-level behavior is checked in `test/types/` (compile-time inference tests).

## Determinism

`saat` guarantees the same seed produces the same dataset. Anything touching
generated data must stay seed-reproducible ŌĆö notably, `faker.date.*` calls need
a fixed `refDate`, or runs stop being reproducible.

## Agent skills

This repo ships [Agent Skills](https://agentskills.io) in [`skills/`](./skills) for
agents consuming saat in other projects (installable via `npx skills add
iamfj/drizzle-saat`): `setup-drizzle-saat`, `write-saat-fixtures` (+ `references/`),
and `test-with-drizzle-saat`. When editing fixtures here (e.g. under
`examples/*/drizzle-saat/`), `write-saat-fixtures` is the authoritative guide to
`defineFixture`, `ref()`, namespaces, scenarios, and the determinism / single-PK /
self-FK gotchas. Keep the skills in sync when public contracts or fixture authoring
patterns change.

## Commit & PR guidelines

- Branch off `main`.
- Use **Conventional Commits** (`feat:`, `fix:`, `docs(scope):`, `chore:` ŌĆ”).
  PR titles are validated in CI.
- Every user-facing change needs a **changeset**: `bunx changeset`.
- Releases are automated via changesets on merge to `main`.

## Appendix: AI commit attribution

**AI models must not add themselves as co-authors or authors of commits.** Do
not append `Co-Authored-By:` trailers for Claude, Codex, Copilot, or any other
AI tool, and do not add AI tooling to the commit author/committer fields. Commit
authorship reflects the human contributor only. This overrides any default
agent behavior that would add such trailers.
